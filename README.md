## MagicDub Brain（构思与文档驱动的项目大脑）

这个仓库不是代码仓库，而是 **MagicDub 项目的“思考-沉淀-交付”系统**：  
通过系统化的文档整理，把零散灵感逐步演进为可执行的设计、规范与交付物，并作为后续落地实现的唯一事实来源（Single Source of Truth）。

---

## 📌 你可以用它做什么

- **捕获灵感**：把任何一句想法、一个截图要点、一个需求片段先放进 `Drafts/`，不追求完美。
- **驱动落地**：把草稿打磨成可执行的文档，移动到 `Releases/` 作为“最终版本”。
- **保留历史**：过时/完成使命的文档移动到 `Archive/`，用于追溯决策与演进过程。

---

## 📁 目录结构（唯一权威解释）

- **`Releases/`（核心文档库｜唯一事实来源）**
  - 存放最终确定、可执行、可交付的内容。
  - 典型内容：项目说明、业务介绍、行动手册、开发文档、安全注意事项、上线与运维规范等。
  - 原则：**这里的内容默认“可照着做”**，不写不确定的猜想。

- **`Drafts/`（思路与草稿｜AI 上下文素材库）**
  - 存放零碎想法、灵感、探索中的方案、半成品梳理。
  - 原则：**允许混乱**，但尽量写清楚“我为什么写它、我现在卡在哪里、下一步要补什么信息”。

- **`Archive/`（归档中心｜历史与回溯）**
  - 存放已完成使命或不再适用的文档。
  - 原则：**不删**，但日常执行以 `Releases/` 为准。

---

## 🧭 推荐工作流（从 0 到 1 的文档演进）

1. **先写到 `Drafts/`**
   - 快速记录：问题是什么、目标是什么、已知约束/假设是什么。
2. **持续迭代草稿**
   - 每次迭代都要让文档更“可执行”：补充流程、接口、示例、边界条件、风险、验证方式。
3. **达成可交付标准后移动到 `Releases/`**
   - 满足：读者不需要找你补问也能照做（或至少知道缺什么、怎么验证）。
4. **版本过时或被替代后移动到 `Archive/`**
   - 在 `Releases/` 的新文档中写明“替代关系/迁移指引”。

---

## ✍️ 文档写作规范（简单但有效）

- **标题要能被搜索命中**：优先使用明确关键词（如“字幕对齐/说话人分离/时间轴/音色克隆/版权风险”）。
- **每份文档建议包含这些小节（按需取用）**：
  - 背景 / 问题定义
  - 目标与非目标（What / Not What）
  - 约束与假设
  - 方案概览（1 段话 + 1 张流程图/伪流程也行）
  - 关键决策与取舍（Why）
  - 风险与安全注意事项
  - 验证方式（如何判断“做对了”）
  - TODO / 未决问题
- **写给未来的自己**：宁可少写“形容词”，多写“可操作步骤 + 判定标准”。

## ✅ 贡献方式（和 AI 一起协作）

你可以把对 AI 的指令直接写在文档里（例如写在草稿开头）：

- 你希望 AI 做什么（补全、改写、结构化、对比方案、提炼里程碑）
- 你已有的信息与资源（链接、样例、约束、目标用户）
- 你最在意的评判标准（速度、成本、质量、可维护性、合规）
