## MagicDub 技术栈与 API（V1 准备清单｜Dashboard 异步）

> 目标：你照着这份清单逐条勾选，就能把 V1 的“全云端 API + Serverless 合成”工作流跑起来。  
> V1 约束：**英→中 + 中→英**、**不做唇形**、**阿里云国际为主**、**dashboard 异步提交/等待**。

---

## 0. 总原则（先读这个）

- **阿里云国际优先**：存储/计算/网络/账单尽量统一在阿里云国际侧。
- **API 可替换**：每个环节只依赖“输入/输出契约（URL + JSON）”，不要把业务写死在某个 SDK。
- **对象存储做中转**：所有中间产物都落到 OSS（便于断点续跑、审计、重试、成本核算）。
- **异步优先**：用户提交后进入队列，dashboard 只展示状态与产物链接；允许长等待来换取更低的资源压力与成本。
- **体验目标**：先做到“译制片观感”（听感自然、BGM 连续、时间轴不乱）；本项目 **永远不做唇形/口型修正**。

## 1. 账号、权限、支付与预算（必须先准备）

- **阿里云国际账号**
  - **要准备什么**：账号、主付款方式、可用区选择原则（靠近目标用户/内容源）。
  - **要检查什么**：是否能正常开通 OSS、函数计算/Serverless、以及 AI 平台能力（ASR/TTS 等）。
- **API Key 管理**
  - **要准备什么**：为“开发/测试/生产”分别创建 Key（不要共用）。
  - **要检查什么**：Key 是否可随时吊销、是否支持子账号/权限最小化。
- **预算与告警**
  - **要准备什么**：日预算、月预算、单任务成本上限（例如“每分钟视频最高成本”）。
  - **要检查什么**：达到阈值是否能自动告警/停用高成本功能（例如跳过某些增强步骤）。

## 2. 云资源（阿里云国际侧）准备清单

### 2.1 OSS（对象存储）

- **Bucket 规划（建议固定前缀/目录）**
  - `raw/`：用户原始视频
  - `stems/`：分离后的 `vocal` / `bgm`
  - `segments/`：切句后的音频片段（若采用）
  - `timeline/`：ASR/对齐/翻译的 JSON
  - `tts/`：生成的人声片段（句子级）
  - `mix/`：混音中间产物（可选）
  - `output/`：最终视频
  - `logs/`：结构化日志（jsonl）
- **上传策略**
  - **要准备什么**：前端直传 OSS（用短时效签名 URL），避免后端转发大文件。
  - **要检查什么**：跨域 CORS、分片上传、失败重传、对象生命周期（自动清理中间产物）。
- **下载与分发**
  - **要准备什么**：最终产物走 CDN 或签名 URL（按你的商业策略决定）。
  - **要检查什么**：是否需要防盗链/过期策略。

### 2.2 计算（两类：编排常驻 + 合成 Serverless）

- **编排服务（常驻/轻量）**
  - **要准备什么**：一个轻量后端服务，用于创建任务、写入状态、调度 API、落盘产物。
  - **要检查什么**：是否具备队列能力（见 2.3），以及与 OSS 的读写权限。
- **合成服务（Serverless + FFmpeg）**
  - **要准备什么**：一个 Serverless 入口（HTTP/消息触发均可），只做“按时间轴合成 + 编码输出”。
  - **要检查什么**：运行时是否带 FFmpeg，或能否把 FFmpeg 打进镜像/层；是否有足够临时磁盘与执行时长上限。

### 2.3 队列与异步（dashboard 的基础）

- **任务队列**
  - **要准备什么**：一个队列/任务系统（只需支持：入队、重试、延迟、死信）。
  - **要检查什么**：最大并发、重试策略、失败告警、任务可观测（任务链路 ID）。
- **状态存储**
  - **要准备什么**：一个数据库/存储记录任务状态（最少 `task_id` + `status` + `progress` + `output_urls`）。
  - **要检查什么**：幂等（重复提交同一视频不会无限生成）、断点续跑（失败后能从最近一步继续）。

## 3. AI API（按主链路）需要准备什么

> 这里不绑定“唯一供应商”，但你可把“阿里云国际 AI 平台能力”作为主干实现。  
> 每个环节都要明确：**输入**（OSS URL/二进制）、**输出**（OSS URL/JSON）、**可重试性**、**计费口径**。

### 3.1 人声 / BGM 分离（必选）

- **目标**：得到 `vocal_stem` 与 `bgm_stem`，保证最终 BGM 连续。
- **输入**：原视频音轨（或从视频提取的音频）。
- **输出**：两条音轨（推荐写回 OSS）。
- **验收点**
  - BGM 连续、无明显断裂
  - vocal 足够干净，便于 ASR（噪声/音乐干扰显著降低）

### 3.2 ASR + 时间戳（必选）

- **目标**：拿到可用于“切句 + 对齐 + dashboard 展示”的时间轴 JSON。
- **输入**：`vocal_stem`（优先）或原音轨（降级）。
- **输出**：至少句子级时间戳；最好包含词级时间戳用于更稳的切句与后续微调。
- **验收点**
  - 支持 **英→中** 场景的英文识别、**中→英** 场景的中文识别
  - 时间戳误差可接受（主观：不抢拍/不拖拍；客观：ms 级统计）

### 3.3 翻译（必选｜需要“长度约束”能力）

- **目标**：不是“翻译得对”就够了，而是要“翻译得对且能在目标时长里读完”。  
  - 英→中：往往更短，需要控制停顿与信息密度  
  - 中→英：往往更长，需要压缩或改写
- **输入**：`src_text` + `start_ms/end_ms`（给翻译模型一个时长约束窗口）。
- **输出**：至少两版：`tgt_text_raw`（直译）与 `tgt_text_constrained`（长度约束版）。
- **验收点**
  - 长度约束可迭代：超长时能“精简但不丢核心信息”
  - 输出稳定：同一句多次调用差异不要过大（便于回归）

### 3.4 TTS / 音色克隆（必选）

- **目标**：生成“像本人在说目标语言”的语音，并可控语速/时长。
- **输入**
  - 参考音频（用于音色克隆，存 OSS）
  - `tgt_text_constrained`
  - 语速/风格参数（如 `speed`/`style`，按你选的服务能力）
- **输出**：句子级人声片段（写回 OSS），并记录实际时长。
- **验收点**
  - 英→中与中→英都能稳定输出
  - 语速可调，并且小幅变速不明显失真

### 3.5 时长校正（对齐兜底，通常放在合成阶段做）

- **目标**：把每句人声塞回 `[start_ms, end_ms]` 窗口。
- **策略（按优先级）**
  - 短了：句尾补静音
  - 长了：轻微变速（控制在小范围）或回到翻译层再压缩
- **验收点**
  - 人耳不明显“变速变调感”
  - 对齐误差可被量化记录

## 4. 合成（Serverless FFmpeg）要准备哪些能力

> 你的 V1 成败，很大概率取决于“合成层做得是否专业”。目标是听起来像电影配音而不是拼贴。

- **必备 FFmpeg 能力清单**
  - 从视频提取音频（如需要）
  - 混音：人声 + BGM（`amix`）
  - 对齐：句子级 offset/delay（`adelay` 或等价方案）
  - 响度标准化：`loudnorm`（避免忽大忽小）
  - 轻微变速：`atempo`（必要时）
  - 编码输出：统一视频/音频编码参数（便于播放兼容）
- **Serverless I/O 约定**
  - **输入**：只接收 OSS URL（原视频、bgm、人声片段、timeline JSON）
  - **输出**：写回 OSS（`output/`），返回下载 URL
- **验收点**
  - BGM 连续无断裂，人声不盖过 BGM（必要时 ducking）
  - 同一 timeline 多次合成结果可复现（便于调参）

## 5. 数据与接口契约（必须先定“骨架”）

> 先定契约，你才能自由换模型/换供应商而不重写全链路。

### 5.1 `SentenceTimelineJSON`（最小字段）

参考 `[Releases/01_MagicDub技术框架.md](Releases/01_MagicDub技术框架.md)`，V1 建议最少包含：

- `sentence_id`
- `start_ms` / `end_ms`
- `src_lang` / `tgt_lang`
- `src_text`
- `tgt_text_raw`
- `tgt_text_constrained`
- `audio_url`（TTS 结果）
- `meta`（建议）：`asr_confidence`、`tts_duration_ms`、`stretch_ratio`、`provider_ids`

### 5.2 任务状态机（dashboard 展示用）

- 主状态：`queued` → `processing` → `ready` / `failed`
- 子状态（建议最少）：`separate` / `asr` / `translate` / `tts` / `mix` / `encode`
- 必备字段（建议）
  - `task_id`、`created_at`、`updated_at`
  - `input_video_url`、`output_video_url`
  - `error_code`/`error_message`（失败时）
  - `cost_estimate`（阶段性累计）

## 6. 可观测、重试与成本控制（V1 也必须有）

- **每一步必须记录**
  - 供应商/模型名（或 provider_id）
  - 请求耗时、返回码、重试次数
  - 产物 URL（写 OSS）
  - 估算成本（至少按“分钟/次”粗略累计）
- **重试策略**
  - 可重试：网络超时、限流、5xx
  - 不可重试：参数错误、鉴权失败、配额不足（需要人工处理）
- **熔断与降级（省钱、省事故）**
  - 超预算：禁止高成本模式（如高音质/高采样率），或拒绝任务
  - 超时：允许跳过非关键步骤（例如声场/混响等增强项）

---

## 7. 安全与合规（V1 最小集）

- **素材与授权记录（建议写入任务元数据）**
  - 来源 URL、上传者、用途声明、是否允许二次分发
- **滥用防护（先最小化）**
  - 任务留痕（谁、何时、处理了什么）
  - 输出可选标识：是否加“AI 翻译配音”提示（先作为开关/策略项记录）

## 8. V1 里你需要“立刻确定”的 5 个选择（写在这里，避免拖延）

> 这些不是让你纠结，是为了让后续实现不反复推倒重来。

1. **对象存储是否作为唯一中转**（推荐：是）
2. **ASR 输出是否要求词级时间戳**（推荐：能要就要）
3. **翻译是否启用长度约束迭代**（推荐：必须）
4. **TTS 是否必须支持语速可控**（推荐：必须）
5. **合成是否统一走 Serverless FFmpeg**（推荐：是，且输入输出一律 OSS URL）
